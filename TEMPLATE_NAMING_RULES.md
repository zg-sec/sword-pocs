# 模板命名与去重规范

## 概述

本文档用于约束POC模板的去重方式、命名规则和后续编写规范。

目标：

- 每个真实漏洞或检测场景只保留一份规范模板
- 文件名与 `id` 保持稳定、可预测、便于搜索
- 避免因为哈希后缀、大小写、描述差异产生重复模板
- 为后续新增模板提供统一的编写与审核标准

## 基本约束

每个模板都必须满足以下条件：

- 文件名去掉扩展名后的 stem 必须与顶层 `id` 完全一致
- 文件扩展名统一使用 `.yaml`
- 非 `CVE` 模板文件名必须为小写 kebab-case
- `CVE` 模板文件名必须以大写 `CVE-` 开头
- 文件名中不得包含空格
- 文件名中不得包含下划线
- 除非本身是有意义的标识，否则不得保留随机哈希后缀

示例：

- 合法：`blind-ssrf-001.yaml`，内部 `id: blind-ssrf-001`
- 合法：`CVE-2021-31800.yaml`，内部 `id: CVE-2021-31800`
- 非法：`0x001.yaml`
- 非法：`Grimag.yaml`
- 非法：`CVE-2022-41082_51pwn.yaml`
- 非法：`telerik-dialoghandler-detect 1.yaml`

## 去重规则

### 1. 完全重复去重

如果两个模板在忽略首行 `id:` 后，其余内容完全一致，则视为完全重复模板。

处理规则：

- 保留更规范、可读性更高的文件名
- 优先保留没有 32 位哈希后缀的版本
- 删除其余重复副本

### 2. 语义重复去重

只有当以下条件同时成立时，两个模板才可以视为语义重复：

- 检测逻辑相同
- `info.name` 相同
- 它们不代表不同的非空 `classification.cve-id`

处理规则：

- 保留一个规范目标文件
- 内容采用信息更完整的那一份
- 将保留文件的 `id` 改为与保留文件名一致
- 删除其余重复副本

### 3. 不允许合并的情况

以下情况不能合并：

- 同一产品，但漏洞不同
- 同一版本判断逻辑，但漏洞语义不同
- 同一插件或主题家族，但 `cve-id` 不同
- 同一产品 slug，但 `info.name` 指向不同利用点或不同漏洞

## 命名规则

### 1. 普通非 CVE 模板

统一使用小写 kebab-case。

格式示例：

- `product-name.yaml`
- `vendor-product-rce.yaml`
- `redirect-redirection-bulk-delete-csrf.yaml`

规则：

- 只允许字母、数字、短横线
- 连续分隔符折叠为一个短横线
- 去掉无意义标点
- 下划线统一转成短横线
- 混合大小写统一转成小写

### 2. CVE 模板

如果存在合法的 `classification.cve-id`，则优先用它作为命名来源。

格式示例：

- 单个模板对应一个 CVE：`CVE-2021-31800.yaml`
- 同一个 CVE 存在多个产品场景：`CVE-2021-44228-apache-druid-log4j-rce.yaml`

规则：

- `CVE` 前缀必须保持大写
- 必须保留完整的 CVE 数字标识
- 同一 CVE 对应多个产品模板时，后面补充可读的产品或场景后缀
- 不要追加无意义编号，优先使用语义后缀

### 3. 占位符或序号模板

如果当前文件名或 `id` 只是占位符，则应从 `info.name` 推导出规范名称。

示例：

- `0x001.yaml` -> `blind-ssrf-001.yaml`
- `0100.yaml` -> `blind-ssrf-100.yaml`

规则：

- 如果原始编号是唯一稳定区分信息，则保留原始编号语义
- 优先采用“描述 slug + 原始序号”
- 不要凭空发明新的随机编号

### 4. 已有语义但不规范的 ID

如果现有 `id` 已经具备明确语义，只是格式不规范，则直接规范化，而不是重新命名成完全不同的新概念。

示例：

- `Grimag.yaml` -> `grimag.yaml`
- `Divi.yaml` -> `divi.yaml`
- `grav_cms.yaml` -> `grav-cms.yaml`

## 目标命名优先级

生成规范文件名和 `id` 时，按以下优先级选择：

1. 合法的 `classification.cve-id`
2. 已有语义明确的 `id`，经规范化后使用
3. `info.name` 转成 kebab-case
4. 对占位式模板，使用 `info.name + 原始数字后缀`
5. 最后才使用清洗后的旧文件 stem 作为兜底

## 冲突处理规则

如果多个模板映射到同一个目标名，处理顺序如下：

1. 先判断它们是否已经是重复模板
2. 如果是重复模板，则保留一份规范文件并删除其余副本
3. 如果不是重复模板，则为目标名补充稳定且可读的后缀

后缀优先级：

1. 当前 `id` 中已有的有意义产品或利用点 slug
2. 从 `info.name` 中提取的可区分语义片段
3. 原始数字后缀
4. 旧语义 stem 中提取出的稳定片段

禁止：

- 把 32 位随机哈希重新带回文件名
- 在存在可读语义后缀时，退化成无意义顺序号

## 新模板编写规范

### 1. 新建前先查重

新增模板前，至少要搜索以下几类信息：

- 相同的 `classification.cve-id`
- 相同或相近的产品 slug
- 相同或高度接近的 `info.name`
- 相同检测逻辑

### 2. 一个模板只表达一个漏洞语义

一个模板应该只表达一个漏洞概念。

不要为了这些原因再创建新模板：

- 只换了一个标题
- 只补充了描述
- 只改了参考链接格式
- 只是在文件名上附加哈希后缀

如果只是同一 CVE 下的不同产品覆盖，应使用可读的 CVE 后缀，而不是复制出无规则模板。

### 3. 保持 `id` 稳定、可搜索、可推断

推荐形式：

- `CVE-YYYY-NNNN`
- `product-vulnerability-slug`
- `description-slug-001`

禁止形式：

- `0x001`
- `0100`
- `test1`
- `foo_final`
- `something-<32位hex>`

### 4. 不要把噪声写进名字

以下信息除非本身就是公开漏洞标识的一部分，否则不要写进文件名或 `id`：

- 随机哈希
- 临时导入编号
- 重复导出痕迹
- 空格
- 下划线
- 非 `CVE` 模板中的混合大小写

## 审核清单

每次新增或修改模板前，至少检查以下项目：

- 文件名 stem 是否等于 `id`
- 命名是否符合规范风格
- 有 `cve-id` 时是否正确使用
- 目录中是否已经存在明显重复模板
- 是否引入了无意义哈希后缀
- 如果名称带编号，该编号是否来源于原始稳定信息

## 推荐维护流程

1. 先按 `cve-id`、产品 slug、`info.name` 搜索现有模板
2. 判断当前工作是新增、变体还是重复
3. 在保存模板前先确定规范文件名
4. 立即让 `id` 与文件名一致
5. 批量导入前先跑一次规范化与去重
6. 发布或提交前跑一遍全量校验

## 校验要求

校验脚本至少应检查以下内容：

- 每个文件的 stem 与内部 `id` 完全一致
- 文件名中不存在空格
- 文件名中不存在下划线
- 非 `CVE` 模板全部为小写 kebab-case
- `CVE` 模板全部符合 `CVE-YYYY-...` 格式
- 不存在重复的规范目标名
- 不存在临时重命名文件

## 变更控制建议

对已有模板做命名规范化时：

- 除非确认是重复模板，否则只改文件名和 `id`
- 不要在做命名清理时顺手修改检测逻辑
- 如果需要从重复模板中合并内容，应保留信息最完整的元数据与参考链接

出现不确定时，优先遵循以下原则：

- 优先保留语义正确性，而不是保留旧字面名字
- 优先使用可读后缀，而不是不透明后缀
- 优先保留一份规范模板，而不是保留多份近似副本
